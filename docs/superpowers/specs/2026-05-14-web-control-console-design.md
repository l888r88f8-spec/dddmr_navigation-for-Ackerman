# Web Control Console Design

## Goal

Build an offline-capable web console that replaces RViz for daily localization
and point-to-point navigation operation.

The console works with the existing launch files:

- `/home/robot/nav26_ws/src/dddmr_navigation-for-Ackerman/src/dddmr_p2p_move_base/launch/p2p_move_base_localization.launch.py`
- `/home/robot/slam_ws/src/lio_sam_hesai/launch/localization_with_nonground.launch.py`

The web console is launched separately. It does not own startup of the
localization or navigation stacks.

## Scope

In scope:

- Display a 2D top-down map for primary operation.
- Display the vehicle's current pose from localization odometry.
- Publish an initial localization pose to `/initialpose`.
- Publish navigation goals through the existing `goal_pose_3d` topic.
- Display point cloud map layers with independent visibility toggles.
- Provide a 3D point cloud debug page for inspection only.
- Run offline without Node, rosbridge, CDN assets, or external web services.

Out of scope:

- Replacing `p2p_move_base` or `lio_sam_hesai` internals.
- Launching the two existing stacks from the web console.
- Subscribing to `/lio_sam/localization/path`.
- Requiring RViz for normal operation.

## Package

Add a new ROS 2 Python package under the navigation workspace:

`nav26_ws/src/dddmr_navigation-for-Ackerman/src/dddmr_web_control`

The package contains:

- `dddmr_web_control/web_control_node.py`: ROS bridge and HTTP server.
- `launch/web_control.launch.py`: starts only the web console.
- `static/`: offline HTML, CSS, and JavaScript assets.
- `test/`: focused unit tests for conversions, point cloud processing, and API
  validation.

The package should use only ROS 2 Python APIs and Python standard library
networking so it works in the current environment.

## ROS Interfaces

Subscriptions:

- `/lio_sam/localization/global_map`
  - Type: `sensor_msgs/msg/PointCloud2`
  - Role: localization map layer.
  - Default visible: true.

- `mapcloud`
  - Type: `sensor_msgs/msg/PointCloud2`
  - Role: navigation global map layer.
  - Default visible: false.
  - Note: this may duplicate `global_map`, so the frontend should not enable
    both by default.

- `mapground`
  - Type: `sensor_msgs/msg/PointCloud2`
  - Role: ground or traversability reference layer.
  - Default visible: false.

- `/lio_sam/localization/odometry`
  - Type: `nav_msgs/msg/Odometry`
  - Role: current vehicle pose.

- Planning/debug path topics published by the navigation stack.
  - Type: `nav_msgs/msg/Path`
  - Role: optional route visualization.
  - The implementation must not subscribe to `/lio_sam/localization/path`.

Publications:

- `/initialpose`
  - Type: `geometry_msgs/msg/PoseWithCovarianceStamped`
  - Frame: `map`
  - Role: initial pose for `lio_sam_hesai_relocalization`.

- `goal_pose_3d`
  - Type: `geometry_msgs/msg/PoseStamped`
  - Frame: `map`
  - Role: goal input consumed by existing `clicked2goal.py`, which forwards to
    the `/p2p_move_base` action.

## Backend Design

`web_control_node.py` runs one `rclpy` node and one local HTTP server in the
same process.

The ROS side caches the latest valid data for each source:

- Point cloud layers are decoded with `sensor_msgs_py.point_cloud2`.
- 2D map data is generated by projecting points to XY and downsampling.
- 3D debug data is downsampled separately to protect browser performance.
- Odometry is cached as position, yaw, frame, and timestamp.
- Optional route paths are cached as ordered XY poses.

The HTTP side serves static files and JSON endpoints:

- `GET /api/state`: source health, latest odom, timestamps, and counts.
- `GET /api/map2d?layer=<name>`: downsampled 2D points for one layer.
- `GET /api/map3d?layer=<name>`: downsampled 3D points for debug view.
- `GET /api/path?topic=<name>`: cached route path data when available.
- `POST /api/initialpose`: validates x, y, z, yaw and publishes
  `/initialpose`.
- `POST /api/goal`: validates x, y, z, yaw and publishes `goal_pose_3d`.

All publish endpoints return JSON with:

- `ok`: boolean.
- `message`: human-readable status.
- `stamp`: ROS time or wall-clock timestamp.

Parameters:

- `host`: HTTP bind host, default `0.0.0.0`.
- `port`: HTTP port, default `8080`.
- `map_frame`: default `map`.
- `global_map_topic`: default `/lio_sam/localization/global_map`.
- `mapcloud_topic`: default `mapcloud`.
- `mapground_topic`: default `mapground`.
- `odom_topic`: default `/lio_sam/localization/odometry`.
- `initialpose_topic`: default `/initialpose`.
- `goal_topic`: default `goal_pose_3d`.
- `path_topics`: list of navigation planning path topics, default empty.
- `max_2d_points_per_layer`: browser safety limit.
- `max_3d_points_per_layer`: browser safety limit.
- `stale_timeout_sec`: source stale threshold.

## Frontend Design

The main screen is a dense operational console, not a landing page.

Layout:

- A full-height 2D canvas occupies the main area.
- A narrow right control panel contains mode, layer, status, and publish
  controls.
- The 3D debug view is a separate tab or view.

2D map behavior:

- Mouse wheel zooms.
- Drag pans in view mode.
- Vehicle is rendered as an oriented arrow or footprint.
- Goal and initial pose previews are rendered before publishing.
- Route paths are rendered when enabled and available.

Modes:

- View: pan and zoom only.
- Initial Pose: click to place position, drag or second click to set heading,
  then publish `/initialpose`.
- Goal: click to place target, drag or second click to set heading, then publish
  `goal_pose_3d`.
- 3D Debug: inspect downsampled point cloud layers.

Layer controls:

- Localization map: `/lio_sam/localization/global_map`, default on.
- Navigation map: `mapcloud`, default off.
- Ground layer: `mapground`, default off.
- Planning path layer, default on when data exists.
- Vehicle layer, default on.

Because `global_map` and `mapcloud` may be the same point cloud, the UI defaults
to showing only `global_map`.

Status controls:

- ROS/backend connection state.
- Source state per layer: `waiting`, `active`, or `stale`.
- Latest odometry age.
- Current x, y, z, yaw.
- Point count per layer.
- Last initial pose publish result.
- Last goal publish result.

## Error Handling

The HTTP server starts even if ROS topics have not published yet.

Each source is tracked independently:

- `waiting`: no valid message received.
- `active`: recent valid message received.
- `stale`: last valid message is older than `stale_timeout_sec`.

Point cloud handling:

- Invalid or unsupported point records are skipped.
- Empty clouds do not replace the last valid cloud.
- Oversized clouds are downsampled before being exposed to the frontend.
- Frame mismatches are reported in status but do not crash the node.

Odometry handling:

- If odometry becomes stale, the vehicle marker turns gray.
- Publishing goals and initial poses remains available while odometry is stale.

Publish handling:

- Reject requests missing x, y, or yaw.
- Convert yaw to quaternion in the backend.
- Publish all commands in the configured `map_frame`.
- Return clear JSON results for both success and rejection.

## Testing

Unit tests:

- yaw-to-quaternion conversion and quaternion-to-yaw display conversion.
- point cloud XY projection and downsampling behavior.
- request validation for `/api/initialpose` and `/api/goal`.
- source health transitions for waiting, active, and stale.

Build and launch checks:

- `colcon build --packages-select dddmr_web_control`
- `ros2 launch dddmr_web_control web_control.launch.py`

Manual integration checks:

- Browser opens the main page at `http://<robot-ip>:8080`.
- Layer toggles show and hide `global_map`, `mapcloud`, and `mapground`.
- Current vehicle pose updates from `/lio_sam/localization/odometry`.
- Initial pose publication appears on `/initialpose`.
- Goal publication appears on `goal_pose_3d` and is accepted by
  `clicked2goal.py`.

## Implementation Decisions

- Planning path subscriptions are configured only through the `path_topics`
  launch parameter. The default is an empty list, so no path topic is subscribed
  until explicitly configured.
- The 3D debug view uses a local basic Canvas/WebGL point renderer shipped in
  this package. It must not depend on CDN assets or Node-based frontend tooling.
- The 2D console is the primary supported workflow. The 3D view is for map and
  coordinate inspection only.
